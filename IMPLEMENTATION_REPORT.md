## 2. Components Implemented

### 2.1 ZK Circuit & Verifier (`zk-circuits/`)

**Circuit:** `BalanceProof` ‚Äî Groth16, 5 public signals  
```
[0] balance_sufficient   ‚Äî 1 iff actual_balance >= required_amount
[1] wallet_commitment    ‚Äî Poseidon(wallet_address, salt)
[2] proof_hash           ‚Äî replay-prevention identifier
[3] required_amount      ‚Äî minimum trade size
[4] timestamp            ‚Äî proof generation time
```

**`verifier-api.ts`** ‚Äî Express server on port 4000:
- `POST /generate-proof` ‚Äî runs `snarkjs groth16 fullProve` with circuit inputs
- ES module compatible (`fileURLToPath` for `__dirname`)
- CORS enabled for frontend
- `toFieldElement()` ‚Äî converts UUID/hex addresses to snark field elements (mod BN254)
- Returns descriptive error JSON on circuit assertion failure (e.g. balance < amount)

**Bug fixed:** UUID nullifiers can't be converted to BigInt directly ‚Äî custom field element encoder added.

---

### 2.2 Smart Contracts (`contracts/`)

**`EscrowVault.sol`**
- `deposit(bytes32 tradeId, address token, uint256 amount, uint256 expiry)` ‚Äî accepts ETH (`token=0x0`) or ERC20
- Escrow key: the raw `tradeId` passed by caller (frontend uses per-party `keccak256(tradeId, address)`)
- `release(bytes32 tradeId, address recipient)` ‚Äî only callable by `OTCSettlement`
- `refund(bytes32 tradeId)` ‚Äî available after expiry, called by depositor
- `getBalance(bytes32 tradeId)` ‚Äî view function used by settlement trigger

**`OTCSettlement.sol`**
- `submitBalanceProof(pA, pB, pC, pubSignals)` ‚Äî full on-chain Groth16 pairing check via `BalanceVerifier`
- `settle(tradeId, buyer, seller, buyerToken, sellerToken, buyerAmount, sellerAmount)` ‚Äî only callable by `creExecutor`
  - Verifies both parties have submitted valid ZK proofs
  - Computes `buyerEscrowId = keccak256(tradeId, buyer)`, `sellerEscrowId = keccak256(tradeId, seller)`
  - Atomically: releases seller's ETH ‚Üí buyer, releases buyer's WLD ‚Üí seller
  - Marks `settlements[tradeId].executed = true`
- `isSettled(bytes32 tradeId)` ‚Äî used by `/api/escrow` to prevent duplicate settlement

Deployed on **Tenderly Ethereum Fork** via `scripts/deploy.js`.

### 2.3 Chainlink CRE Workflow (`privotc-cre/my-workflow/`)

**`privotc-workflow.ts`** ‚Äî trigger: `cron-trigger@1.0.0`

**Handler: `handleFetchFromFrontend`**
1. `GET /api/trade` ‚Äî pulls all pending trades from frontend queue
2. For each trade:
   - `validateWorldId()` ‚Äî deduplicates nullifiers per wallet address (`wallet:0x...` fallback for IDKit staging proofs that lack `nullifier_hash`)
   - ZK proof structure validation (in simulation mode, trusts frontend on-chain verification)
   - Adds to in-memory orderbook
3. `runMatchingEngine()` ‚Äî crosses buy/sell orders on `ETH/WLD` pair (price overlap check)
4. For each match:
   - Prepares on-chain `settle()` calldata (simulation mode: logs only)
   - POSTs match to `POST /api/matches` with **base64-encoded body** (required by CRE protobuf HTTP capability)
   - Match payload: `{ matchId, tradeId, buyerAddress, sellerAddress, ethAmount, wldAmount, deadline }`

**Key bugs fixed:**
- CRE HTTP `body` field must be base64 (`Buffer.from(json).toString('base64')`) ‚Äî not raw string
- Nullifier collision between two wallets using IDKit staging ‚Üí wallet-address-based fallback
- Orderbook depth check updated to `ETH/WLD` pair

### 2.4 Frontend API Routes (`frontend/app/api/`)

**`/api/trade`**
- `POST` ‚Äî validates and enqueues trade (World ID proof + ZK proof + on-chain tx required)
- `GET` ‚Äî returns pending trades; drains queue on read (CRE consumes them)
- Allowed pairs: `ETH/WLD` only

**`/api/matches`** *(new)*
- `GET ?wallet=0x...` ‚Äî returns active (non-settled, non-expired) matches for a wallet
- `POST` ‚Äî stores a match from CRE (called after matching engine runs)
- `PATCH { matchId }` ‚Äî marks a match as settled
- `DELETE` ‚Äî clears all matches (used to reset stale state)
- `Match` interface exported and consumed by `EscrowDeposit` component and `trade/page.tsx`

**`/api/escrow`** *(new)*
- `POST { matchId, tradeId, buyerAddress, sellerAddress, ethAmount, wldAmount }`
- Reads on-chain `EscrowVault.getBalance(buyerEscrowId)` and `getBalance(sellerEscrowId)`
- When both > 0: re-checks `isSettled` on-chain (prevents race condition from concurrent polls), then calls `OTCSettlement.settle()` via deployer wallet
- Catches `SettlementAlreadyExecuted` (error `0x94ce95d2`) ‚Äî returns `{ settled: true }` gracefully
- Uses `viem` `privateKeyToAccount` with deployer key

### 2.5 Frontend Components (`frontend/`)

**`components/privotc/EscrowDeposit.tsx`** *(new)*

Full deposit UI displayed after CRE posts a match:
- Shows: pair, ETH amount, WLD amount, role (BUYER/SELLER), counterparty address, match ID
- 30-minute countdown badge (turns red under 60s, shows EXPIRED at 0)
- **Buyer flow:** `ERC20.approve(escrow, wldAmount)` ‚Üí `EscrowVault.deposit(myEscrowId, WLD, wldAmount, deadline)`
- **Seller flow:** `EscrowVault.deposit(myEscrowId, 0x0, ethAmount, deadline)` with `value: ethAmount`
- Uses **per-party escrow ID**: `keccak256(encodePacked(['bytes32','address'], [tradeId, address]))` ‚Äî matches what `OTCSettlement.settle()` looks up
- Polls `POST /api/escrow` every 5s while in `waiting` state
- Steps: `idle ‚Üí approving ‚Üí depositing ‚Üí waiting ‚Üí settled`
- Receipt status check: throws if `receipt.status === 'reverted'`

**Bug fixed:** Originally both buyer and seller deposited using raw `match.tradeId` ‚Üí second deposit reverted with `EscrowAlreadyExists`. Fixed by using `myEscrowId` (per-party hash) as the escrow key.

**`components/privotc/OrderForm.tsx`** *(modified)*
- Better ZK error messages: detects "Assert Failed"/"BalanceProof" in 500 response ‚Üí shows human-readable message with wallet balance and required amount

**`app/trade/page.tsx`** *(modified)*
- `activeMatch` state, `pollMatches` ‚Äî polls `GET /api/matches?wallet=` every 5s after trade submitted
- Renders `<EscrowDeposit match={activeMatch}>` below the order form when a match exists

## 3. End-to-End Flow (Verified Working)

```
1. User visits /verify ‚Üí World ID IDKit popup ‚Üí proof stored in sessionStorage

2. User fills order form (ETH/WLD, amount, price, BUY/SELL)

3. [Frontend] Fetches on-chain ETH balance from Tenderly RPC

4. [Frontend] POST http://localhost:4000/generate-proof
   ‚Üí snarkjs groth16 fullProve with balance & amount inputs
   ‚Üí Returns { proof: {pi_a,pi_b,pi_c}, publicSignals: [5] }
   ‚Üí publicSignals[0] must be "1" (balance sufficient)

5. [Frontend] OTCSettlement.submitBalanceProof(pA, pB, pC, pubSignals)
   ‚Üí Full Groth16 pairing check on-chain
   ‚Üí Emits BalanceZKVerified event
   ‚Üí Stores balanceVerified[msg.sender] = true

6. [Frontend] POST /api/trade
   ‚Üí Trade queued with World ID proof + ZK proof + on-chain tx hash

7. [CRE] cron-trigger fires ‚Üí GET /api/trade ‚Üí pulls queue
   ‚Üí validateWorldId() per trade
   ‚Üí runMatchingEngine() on ETH/WLD orderbook
   ‚Üí finds 1 match (buy @ 15, sell @ 15)
   ‚Üí POST /api/matches (base64 body) with 30-min deadline

8. [Frontend] Both tabs polling GET /api/matches ‚Üí detect match
   ‚Üí EscrowDeposit UI appears simultaneously on both tabs

9. [Buyer tab] Approve 150 WLD ‚Üí deposit 150 WLD to escrow
   [Seller tab] Deposit 10 ETH to escrow

10. [Frontend] POST /api/escrow (polls every 5s)
    ‚Üí Reads EscrowVault.getBalance(buyerEscrowId) = 150 WLD ‚úÖ
    ‚Üí Reads EscrowVault.getBalance(sellerEscrowId) = 10 ETH  ‚úÖ
    ‚Üí OTCSettlement.settle() sent by deployer wallet
    ‚Üí EscrowVault.release(sellerEscrowId ‚Üí buyer)  // buyer gets 10 ETH
    ‚Üí EscrowVault.release(buyerEscrowId  ‚Üí seller) // seller gets 150 WLD
    ‚Üí TradeSettled event emitted on-chain

11. EscrowDeposit UI ‚Üí "üéâ Trade Complete"
    Both page.tsx ‚Üí "Trade settled!"
```

## 5. Bugs Fixed During Implementation

| # | Bug | Fix |
|---|---|---|
| 1 | `__dirname` undefined in ES module | `fileURLToPath(import.meta.url)` + `path.dirname` |
| 2 | CORS blocking ZK verifier requests | Added CORS middleware before `express.json()` |
| 3 | UUID nullifier can't convert to BigInt | `toFieldElement()` ‚Äî byte accumulation mod BN254 field |
| 4 | `nullifier_hash: undefined` ‚Üí API 400 | Relaxed `/api/trade` check to just `typeof proof === 'object'` |
| 5 | Both wallets get same nullifier | Changed fallback to `wallet:${walletAddress}` (unique per wallet) |
| 6 | CRE HTTP body must be base64 | `Buffer.from(json).toString('base64')` before `sendRequest` |
| 7 | Both parties deposit to same escrow slot | Use `keccak256(tradeId, address)` per-party ID in deposit calls |
| 8 | `waitForTransactionReceipt` doesn't throw on revert | Check `receipt.status === 'reverted'` explicitly |
| 9 | Concurrent `/api/escrow` polls cause duplicate `settle()` | Re-read `isSettled` before writing; catch `0x94ce95d2` as success |
| 10 | Trade queue drained by status checks | Added `?drain=false` flag; only CRE drains on GET |
| 11 | 5-minute escrow deadline too short | Extended to 30 minutes |
