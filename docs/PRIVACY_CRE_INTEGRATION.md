- **Wallet balances** visible to all participants
- **Order details** in public mempools before execution
- **Trading strategies** revealed through on-chain patterns


## PrivOTC Privacy Architecture

### Layer 1: Zero-Knowledge Balance Proofs

**Problem:** How to prove you have funds without revealing balance?

**Solution:** ZK circuits using Circom + Groth16

```circom
// balance_proof.circom
pragma circom 2.0.0;

include "circomlib/comparators.circom";
include "circomlib/poseidon.circom";

template BalanceProof() {
    // PRIVATE INPUTS (never revealed)
    signal input balance;         // User's actual balance (secret)
    signal input salt;            // Random salt (prevents replay)
    
    // PUBLIC INPUTS (visible on-chain)
    signal input minRequired;     // Minimum balance needed
    signal input commitment;      // Hash of balance + salt
    
    // OUTPUT
    signal output valid;          // 1 if proof valid, 0 otherwise
    
    // Constraint 1: Verify balance commitment
    component hasher = Poseidon(2);
    hasher.inputs[0] <== balance;
    hasher.inputs[1] <== salt;
    commitment === hasher.out;
    
    // Constraint 2: Prove balance >= minRequired
    component gte = GreaterEqThan(64);
    gte.in[0] <== balance;
    gte.in[1] <== minRequired;
    valid <== gte.out;
}

component main {public [minRequired, commitment]} = BalanceProof();
```

**What This Proves:**
- ✅ User has balance ≥ minRequired
- ✅ Balance commitment is valid
- ❌ DOES NOT REVEAL: Actual balance amount
- ❌ DOES NOT REVEAL: Salt used

**On-Chain Verification:**

```solidity
// BalanceVerifier.sol
contract BalanceVerifier {
    mapping(bytes32 => bool) public validProofs;
    
    function verifyProof(
        uint[2] memory a,
        uint[2][2] memory b,
        uint[2] memory c,
        uint[2] memory input  // [minRequired, commitment]
    ) public view returns (bool) {
        // Groth16 verification (generated by Circom)
        return this.verifyProof(a, b, c, input);
    }
    
    function registerProof(
        address user,
        bytes32 proofHash,
        uint[2] memory a,
        uint[2][2] memory b,
        uint[2] memory c,
        uint[2] memory input
    ) external {
        require(verifyProof(a, b, c, input), "Invalid ZK proof");
        
        validProofs[proofHash] = true;
        emit ProofRegistered(user, proofHash);
        
        // BALANCE NEVER REVEALED ON-CHAIN
    }
}
```

```typescript
async function confidentialMatching() {

  // Step 5: Prepare settlement data (encrypt sensitive info)
  const settlements = matches.map(match => {

    const tradeId = keccak256(encodePacked(
      ['address', 'address', 'uint256'],
      [match.buyer, match.seller, match.timestamp]
    ))
    
    return {
      tradeId,
      buyerAddress: match.buyer,
      sellerAddress: match.seller,
      chain: match.chain,
      token: match.token,
      // Amounts encrypted until settlement
      encryptedDetails: encryptInTEE({
        ethAmount: match.ethAmount,
        wldAmount: match.wldAmount,
        price: match.price
      }),
      deadline: Math.floor(Date.now() / 1000) + 3600 // 1 hour
    }
  })
  
  // Step 6: Publish matches (addresses public, details private)
  await httpRequest({
    url: 'https://privotc.com/api/matches',
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ matches: settlements })
  })
```

### Layer 4: Privacy-Preserving Settlement

**Problem:** How to settle without revealing wallet balances?

**Solution:** Escrow + ZK proofs + Atomic swaps

**Settlement Flow:**

```
1. ESCROW DEPOSIT (With ZK Proof)
   ├─ Buyer deposits funds to EscrowVault
   ├─ Deposit amount visible (required for escrow)
   ├─ BUT: Original balance still hidden (ZK proof)
   └─ Seller deposits funds to EscrowVault
   
2. BALANCE VERIFICATION (Zero-Knowledge)
   ├─ Contract checks ZK proof validity
   ├─ Proves: balance >= deposit amount
   ├─ Does NOT reveal: Total wallet balance
   └─ Allows: Settlement only if proofs valid for BOTH parties
   
3. ATOMIC SETTLEMENT
   ├─ OTCSettlement.sol executes swap
   ├─ Buyer receives seller's deposit
   ├─ Seller receives buyer's deposit
   ├─ All or nothing (atomic)
   └─ Privacy preserved: No balance history exposed
```

**Settlement Contract:**

```solidity
// OTCSettlement.sol
contract OTCSettlement {
    EscrowVault public escrow;
    BalanceVerifier public verifier;
    
    mapping(bytes32 => bool) public settled;
    
    function settle(
        bytes32 tradeId,
        address token,
        uint256[] calldata amounts,
        address[] calldata parties
    ) external onlyCREExecutor {
        require(!settled[tradeId], "Already settled");
        
        // Verify BOTH parties have valid ZK proofs
        bytes32 buyerEscrowId = keccak256(abi.encodePacked(tradeId, parties[0]));
        bytes32 sellerEscrowId = keccak256(abi.encodePacked(tradeId, parties[1]));
        
        require(
            verifier.hasValidProof(parties[0], buyerEscrowId),
            "BalanceProofRequired: Buyer"
        );
        require(
            verifier.hasValidProof(parties[1], sellerEscrowId),
            "BalanceProofRequired: Seller"
        );
        
        // Execute atomic swap (privacy-preserving)
        escrow.release(buyerEscrowId, parties[1], token, amounts[0]);
        escrow.release(sellerEscrowId, parties[0], token, amounts[1]);
        
        settled[tradeId] = true;
        
        emit Settled(tradeId, parties[0], parties[1]);
        // AMOUNTS NOT IN EVENT (privacy preserved)
    }
}
```

**What's On-Chain:**
- ✅ Trade settled: Yes/No
- ✅ Parties involved: Addresses
- ✅ ZK proof validity: Verified
- ❌ Original wallet balances: HIDDEN
- ❌ Profit/loss: NOT REVEALED
- ❌ Trading strategy: UNKNOWN
